# Brainstorming

## Current Goal

- 定义 `SimplePool` 的首版范围：一个单机自托管的简单连接池系统，管理代理节点、动态分组和 HTTP 代理隧道。
- 明确后端采用 `Go + Gin + SQLite + sing-box`，前端采用 `React + shadcn/ui`，前端面板占用端口 `7891`。
- 将当前想法收敛到可落地的产品边界，避免后续实现阶段架构漂移。

## Problem Definition

- 需要解决什么：
  - 统一管理代理节点。
  - 用正则规则把节点自动组织成组。
  - 从组中创建 HTTP 代理隧道，并基于延迟测试锁定一个可用节点。
  - 通过 `sing-box` 承载节点代理、订阅导入、端口监听和请求转发。
- 为什么现在重要：
  - 这是一个全新项目，若不先压实边界，后续会在“自动切换 / 多用户 / 全协议兼容 / 多机部署”等方向上无序膨胀。
- 如果不先定义清楚会怎样：
  - 数据模型会反复推翻。
  - `sing-box` 控制方式可能选错，导致后续无法无重启刷新。
  - 前后端接口会因为职责不清反复改动。

## Confirmed Facts

- 项目名为 `SimplePool`。
- 首版定位是单管理员系统，不做多租户。
- 首版部署方式是单机自托管，后端、SQLite、`sing-box`、前端都运行在同一台机器上。
- 节点需要支持增加、删除、查询。
- 节点来源包含：
  - 手动新增节点。
  - 导入节点。
  - 订阅链接导入。
- 节点导入边界为“常见订阅 + 常见协议”，不追求首版完整兼容所有 `sing-box` 配置形态。
- 组需要支持 CRUD。
- 组的主模型是“动态正则组”：
  - 创建组时可传入 `filter`。
  - `filter` 基于节点名正则匹配。
  - 组成员随节点集合变化而自动变化。
- 隧道需要支持 CRUD 和刷新。
- 隧道从组创建，本质是一个 HTTP 代理。
- 创建隧道时需要：
  - 从组中通过延迟测试选出一个可用节点。
  - 锁定该节点作为当前使用节点。
  - 分配一个随机可用端口。
- 一个隧道对应一个 `sing-box` 实例。
- 隧道实例启动时会把整个组内节点都装入该实例，而不是只装当前锁定节点。
- 刷新隧道时：
  - 重新从组中找一个可用节点并锁定。
  - 目标是不重新创建新实例，而是在现有实例内完成切换。
- 隧道切换策略以“手动刷新”作为主路径，不做默认自动优选。
- 组成员变化对已运行隧道采用“创建时快照”语义：
  - 隧道只在创建或刷新时读取组成员。
  - 运行中的隧道不自动同步组变化。
- 隧道运行中若当前节点失效，首版行为是标记异常并等待用户手动刷新，不自动切换。
- 隧道刷新时若组内无可用节点：
  - 保留旧节点和当前运行状态。
  - 返回错误并记录失败原因。
- 订阅刷新合并策略为“按来源 + 名称覆盖”。
- 后台管理面板采用固定账号密码 + 会话鉴权。
- 隧道默认不要求认证，但在创建隧道时可以可选传入认证信息。
- 隧道协议范围首版明确只支持 HTTP。
- 节点可用性判断采用固定测试 URL，在超时阈值内选择最低延迟节点。
- 前端需要覆盖以下能力：
  - 添加节点。
  - 导入节点 / 订阅链接。
  - 展示节点。
  - 节点延迟测试。
  - 组操作。
  - 隧道操作。
- 界面风格可参考当前目录 `image.png`：偏控制台式布局，左侧列表，右侧详情与统计卡片。

## Constraints and Boundaries

- 时间：
  - 目标是先落首版，不做过度预埋。
- 预算：
  - 默认低成本自托管方案，不引入重型基础设施。
- Technology / Platform：
  - 后端：`Go`、`Gin`、`SQLite`、`sing-box`
  - 前端：`React`、`shadcn/ui`
  - 隧道类型：HTTP 代理
  - 前端端口：`7891`
- Team / Resources：
  - 按单人维护的复杂度控制范围。
- Compliance / Policy：
  - 当前未定义额外合规要求，但需要注意订阅地址、代理认证信息、后台密码的敏感数据存储与展示。
- Explicit non-goals：
  - 多租户
  - 多机调度
  - 高可用集群
  - 首版 SOCKS5 隧道
  - 首版自动故障转移
  - 首版定时自动切优
  - 首版完整 `sing-box` 全量配置导入
  - 首版以静态手选组为主模型

## Explored Directions

1. 隧道切换策略
   - 方向一：手动刷新切换
   - 方向二：故障自动切换
   - 方向三：定时自动优选
   - 当前结论：选方向一。原因是行为可预测，状态模型简单，便于先把 `sing-box` 控制链路做稳。

2. 组模型
   - 方向一：动态正则组
   - 方向二：静态手选组
   - 方向三：动静态混合
   - 当前结论：选方向一。原因是贴合“根据节点名自动形成组”的核心理念，减少人工维护。

3. 鉴权边界
   - 方向一：后台固定账号密码 + 会话；隧道默认无认证，可按隧道可选配置认证
   - 方向二：后台 HTTP Basic Auth；隧道全局统一认证
   - 方向三：无鉴权
   - 当前结论：选方向一。原因是足够简单，同时保留隧道级共享能力。

4. 节点导入范围
   - 方向一：常见订阅 + 常见协议
   - 方向二：完整 `sing-box` 导入兼容
   - 方向三：只支持手动录入
   - 当前结论：选方向一。原因是首版价值最高，避免把时间耗在解析器兼容泥潭里。

## Rejected or Deferred Directions

1. 多用户 / 多租户平台化设计，延期。
2. 多机、多实例编排，延期。
3. 隧道故障自动切换，延期。
4. 周期性自动测延迟并自动切优，延期。
5. 完整 `sing-box` 任意配置导入，延期。
6. 动静态混合组模型，延期。
7. 纯 HTTP Basic Auth 后台方案，不采用为主路径。

## Assumptions

- `sing-box` 存在可行的“同实例内切换当前节点”控制路径，否则“刷新不重建实例”这一目标需要调整。
- SQLite 足以承载首版配置与状态存储。
- HTTP 代理已足够覆盖首版使用场景，不需要首版同步提供 SOCKS5。
- 节点延迟测试可以由后端或 `sing-box` 能力完成，不需要额外引入外部探测服务。
- 参考图只提供布局与信息架构方向，不要求像素级复刻。
- 测试策略以“后端单元 + 集成优先”为主，前端先做基础交互验证。
- 验收重点是隧道生命周期正确。
- 首版核心失败路径至少覆盖：
  - 无可用节点 / 刷新失败
  - 订阅导入异常
  - 鉴权会话异常
- 性能目标按小规模场景控制：几十到几百节点、少量隧道。

## Risks and Unknowns

- `sing-box` 的控制方式是本项目最大技术风险：
  - 是否能在不重启实例的前提下切换当前使用节点，需要实证验证。
- 延迟测试机制尚未定义：
  - 使用何种测试目标、超时阈值、并发策略、失败判定，还未确认。
- 订阅解析与去重存在协议差异风险：
  - “来源 + 名称覆盖”简单，但不同订阅源可能存在同名冲突、字段不稳定等问题。
- 随机端口分配存在进程崩溃后的回收问题。
- 隧道可选认证信息的存储方式需要谨慎处理，避免明文泄漏风险。

## Open Questions

1. 隧道刷新具体依赖 `sing-box` 的哪种控制方式实现：`selector`、外部控制 API、还是其它机制？
2. 节点延迟测试的技术方案是什么：测试 URL、超时、频率、结果缓存策略如何定义？
3. 订阅源的唯一标识和手动节点的去重规则如何定义，才能避免“来源 + 名称覆盖”带来的误判？

## Recommended Next Steps

1. 先做一轮技术验证，确认 `sing-box` 是否支持“单实例装载整组节点 + 运行中切换选中节点”。
2. 基于当前边界输出系统设计文档，至少覆盖：
   - 节点、组、隧道、订阅源的数据模型
   - `sing-box` 实例管理器
   - 端口分配器
   - 延迟测试流程
   - 认证与会话模型
3. 明确 API 草案与前端页面信息架构，再进入正式开发。
4. 在实现前补一轮专项决策：
   - 组成员变更如何影响运行中隧道
   - 延迟测试的判定标准
   - 可选隧道认证的存储方案
